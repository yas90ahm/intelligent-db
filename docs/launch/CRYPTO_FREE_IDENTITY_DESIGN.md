# Crypto-Free Identity/Independence Layer for Intelligent DB — Design Synthesis

*Panel: researcher, DB expert, senior dev, operator. Synthesized into one design.*

---

## 1. Problem Restated

Intelligent DB exists to fix two failure modes of LLM agents: they **forget** (nothing survives outside the context window) and they **hallucinate** (they cannot distinguish genuine recall from invention). The spider-web memory model — latent strands, activated by spreading activation, never queried by fuzzy nearest-neighbor — solves forgetting-with-structure. The Source-Identity Layer exists to solve the harder half: telling genuine corroboration from manufactured agreement.

The project's own proven theorem is why this layer cannot be dropped just because "identity is priced, not prevented" no longer needs literal pricing: **no purely internal graph rule can simultaneously let one true witness overturn a planted false claim and stop two fake witnesses from overturning a true incumbent**, once minting a new apparent identity has *any* nonzero, attacker-payable cost. Removing Ed25519/DNS-01/Merkle/staking does not remove the theorem — it removes one particular (expensive, self-built) way of pricing identity. The owner's deployment reality changes *who* the attacker is, not *whether* the theorem applies:

- **Inside the owner's own environment** (one company's agent fleet, one person's agents), there genuinely is no adversary to price — agents are cooperative by default. But they are not *epistemically* trustworthy by default: agent A telling agent B a finding, and B writing it to memory, is a **relay**, not a second causal root, and if the independence layer can't tell the difference it will manufacture corroboration for free, with zero attacker required. This is a structural bug, not a Sybil attack, and it needs a structural fix regardless of cryptography.
- **Outside the boundary** (scraped pages, search results, unaffiliated documents) the original adversarial premise is fully intact: a patient attacker can still stand up many URLs, many "publishers," many search-result pages at near-zero cost. The theorem's requirement — a **structural, externally-rooted, costly-to-fake-at-volume signal of independence** — still applies exactly as before; only the *mechanism* that supplies that signal changes, from a cryptographic anchor this project built and signed, to a claim this project *consumes* from infrastructure it doesn't own (an IdP's SSO claims, a domain's registration/operator record).

So the redesign's job is precise: keep the *shape* of "independence is priced, not declared," delete every place this project was the one doing the pricing/signing/attesting, and add exactly one new mechanism (causal-origin-rooted independence) the crypto era didn't need because it was solving a different, adversarial-outsider-shaped problem.

## 2. What Is Preserved, Unconditionally

Nothing about the memory model, the two invariants, or the demote-never-delete contract is in scope for this redesign, and — critically, confirmed independently by the DB expert and senior dev by reading the actual source — most of the *identity math* is already mechanism-agnostic and needs **zero code changes**:

- **`identity/anchors.ts`** — `independenceBetween`, `combineSublinear`, `applySelfStackCap`, `repCapFor`, `stakeIndependenceWeight`. Confirmed to import nothing but `core/types.ts` and operate purely over `AnchorBinding[]` (`{anchorClass, realizedCost, independenceWeight}`). It has never known or cared whether a binding came from a signed attestation or a plain claim.
- **`identity/reputation.ts`** — the Beta(α,β) ledger (`createReputationLedger` / `createSqliteReputationLedger`), independence-weighted α, 4× asymmetric β, decay-on-read, LCB readout. Consumes only an injected weight and `repCapOf(sourceId)`; never touches keys or signatures.
- **`identity/index.ts`**'s facade logic — `stampFor`, the Bron–Kerbosch exact `independentRootCount`, and `independentSources` (the RC-5 fail-closed distinct-approver predicate). These sit behind `KeyRegistryPort` / `AnchorRegistryPort` / `ReputationLedgerPort` interfaces already; a crypto-free registry just has to satisfy the same port shapes.
- **`core/types.ts`** — `AnchorClass`, `AnchorBinding`, `IdentityStamp`, `ProvenanceRoot`, `Edge`, `Strand` are already mechanism-agnostic data shapes. (One additive field is proposed in §4, not a change to any existing one.)
- **`forgetting/consolidation.ts`** (`tryConsolidate`) — consumes `stampsByRoot: Map<ProvenanceRootId, IdentityStamp>` and plain numbers. The single-class-echo tiebreak, the decisive-or-defer multi-class gate (`decisiveMargin = 0.30`, `minWinnerReputation = 0.20`), and the high-impact gate are untouched.
- **`ratification/corroboration.ts`** and the graph-walking core of **`ratification/disown.ts`** (the BFS over `store.inEdges` filtered to `EdgeType.DERIVATION`, the exact `reverseCredit`) — both operate on `StrandId`/`SourceId`/timestamps and store edges, no crypto anywhere.
- The **strand data model**: `LIVE`/`PROVISIONAL`/`DEMOTED`, observed-vs-derived, demote-never-delete, the archive stub, corroboration-vs-echo — none of this is identity/crypto machinery; it is the memory model itself and is explicitly out of scope.

This large untouched core is the main reason the migration is tractable at all (§7): the load-bearing *math* was never the crypto.

## 3. What Is Removed, and Why It's Safe

Every removal is justified the same way: **what independence-pricing work was this mechanism actually doing, and does removing it cost that work or not?**

| Removed | What it actually contributed | Why removal is safe |
|---|---|---|
| **Ed25519 passports** (`identity/keys.ts`) | Proof of *sameness* (same key ⇒ same source), for echo-collapse. | A bare key already contributes `independence_weight = 0.00` by design — passport crypto was doing zero Sybil-pricing work, only "is this the same actor," which a session/account identity already provides for free. |
| **DNS-01 / email-round-trip binders** (`identity/binding.ts`, `identity/anchorRegistry.ts`'s `verifyAttestation`) | Cryptographic proof that a claimed domain/inbox was really controlled, feeding the DOMAIN/EMAIL anchor rows. | The *pricing* logic those rows encode (registrar-throttled cost, rate-limited inboxes) is unaffected by *who checks* the proof. An SSO IdP or a plain domain-of-origin/WHOIS lookup supplies an equivalent-strength claim without this codebase re-implementing DNS-01. |
| **KYC / hardware attestation / financial stake rows** | Reserved but never built (CLAUDE.md's own gap list already lists these as unimplemented). | Nothing to remove — they were never shipped. Their absence stays an honest, named gap, not a regression. |
| **RFC-6962 Merkle log + signed epoch tree heads + independent witness publication** (`ratification/merkleLog.ts`) | Efficient O(log n) third-party inclusion/consistency proofs, detecting rollback/deletion *given an honest published anchor and live witnesses*. | Per CLAUDE.md's own gap list, real external witness sinks were **never built** — "the Merkle guarantee is detection-given-an-honest-published-anchor, null without live witnesses." This machinery was not load-bearing in production today; removing it forecloses a capability the project might want later (efficient third-party proof verification) but costs nothing it was currently delivering. |
| **Ed25519 signing on the ratification ledger** (`ratification/pendingLedger.ts`) | Cryptographic non-repudiation of *who* authored a PENDING/APPROVAL record, verifiable **offline, independent of the writing process** — a check that survives even a compromised DB-writing process. | **This is not a free removal** — see §5's honest disclosure. The bare SHA-256 hash-chain survives (it's a checksum, not a PKI system, and is kept), but attribution now rests on the writing process's own say-so at write time. This is a real, named security regression, not a costless one. |
| **On-chain settlement / staking** | Reserved, never built. | Same as KYC — nothing to remove. |

The throughline, per the DB expert and senior dev's shared framing: crypto that was pricing *sameness* (passports) cost nothing to remove because sameness never priced independence. Crypto that was pricing *external scarcity* (DNS, KYC) is replaced by consuming an equally-scarce external claim instead of manufacturing proof of it. Crypto that was providing *non-repudiation against this project's own compromised process* (ledger signing) is the one case where something real is actually given up, and it is disclosed as such rather than waved away.

## 4. The New Model

### 4.1 The crypto-free channel/tier lattice

Three tiers of `AnchorBinding` producer, all feeding the *existing* `AnchorBinding{anchorClass, realizedCost, independenceWeight}` shape — no new consumer-side code required, only new producers:

**Tier 1 — Home/SSO-verified.** `sourceId = stableHash(idpIssuer, subjectClaim)` — the OIDC `sub` or SAML `NameID` an app's own auth middleware already verified before this project's code runs. A first-party agent process gets its `sourceId` the same way, from its own SCIM-provisioned service-account/workload-identity claim — no keypair minted by this project either way. `operatorClassId` = the IdP **tenant/org id**, not the individual, so two employees (or two agents) inside one company's SSO never manufacture independent roots just by being different people — directly implementing the owner's own framing: *"if a company is using it for their agents, the normal everyday folks are within the same SSO."*

**Tier 1 calibration — the panel's one real internal disagreement, resolved.** The DB expert and researcher initially proposed mapping any SSO group/role claim directly onto the existing `ORGANIZATION` (0.75/0.92) or `VERIFIED_HUMAN` (0.70/0.90) rows with "no policy-number change." The senior dev's review correctly flags this as a category error: those rows price *HR-verified onboarding a company paid for*, but the Sybil-relevant boundary is **cross-tenant** independence — and spinning up a brand-new SSO tenant (Okta/Auth0/Google Workspace) with a disposable email and no verified custom domain is a five-minute, near-zero-cost, self-service action, cheaper than the existing `EMAIL_OAUTH` row (0.10/0.30). Reusing `ORGANIZATION`-grade weight for a bare fresh tenant would silently reintroduce the contradiction-set-bomb the whole layer exists to defeat, under a friendlier name. **Resolution adopted here:** split into two rows —
- `SSO_TENANT_MEMBER` (weight ≈0.10–0.15, repCap ≈0.25–0.30 — EMAIL_OAUTH/PHONE_SIM grade): the default for "belongs to some distinct tenant, no further verification."
- Full `DOMAIN` (0.35) or `ORGANIZATION`-grade weight is reserved for a tenant that has bound a **verified custom domain** to its SSO config (converging back onto the existing, already-priced `DOMAIN` binder machinery) or a real, independently-verified org registration — deliberately *not* reachable by a bare tenant claim alone.

This is the one place the final design departs from the DB expert's and researcher's initial proposal, and it is adopted here because it's the difference between the redesign genuinely preserving the hard theorem's guarantee and only appearing to.

**Tier 2 — Curated-external (research content).** New anchor rows, additive to `AnchorClass`, deliberately priced **below** the rows they superficially resemble:
- `PUBLISHER_UNVERIFIED` (weight ≈0.03–0.05, repCap ≈0.08–0.10): default for any fetched/scraped content whose eTLD+1 has no track record — below `EMAIL_OAUTH`, "some page said so."
- `PUBLISHER_TRACKED` (weight ≈0.15–0.20, repCap ≈0.30–0.35): a synthetic `SourceId = publisher:${eTLD+1}` keyed into the **existing, unmodified** Beta reputation ledger — a publisher earns this tier the same way any source earns reputation, over real tenure, decayed on read, capped. Deliberately kept below `DOMAIN`'s 0.60 repCap so that "a publisher, however old" stays structurally weaker than "a domain someone actually registers and controls" — because the LCB readout *is* reachable given enough tenure (confirmed from `reputation.ts`'s decay/ratify semantics: capped, not artificially unreachable), the repCap value is genuinely load-bearing, not cosmetic.
- `operatorClassId` for both rows reuses the **existing** registrar/ASN operator-cluster lookup the `DOMAIN` binder already has (`identity/binding.ts`) — this is what makes N syndicated URLs behind one registrar collapse to ~1 independent root, directly answering the owner's "many pieces of content from a single publisher... by volume" requirement, for free, with no new ownership-cluster table needed at MVP. A NewsGuard-style curated-ownership table (researcher's proposal) is a defensible v2 refinement, deferred because the ASN/registrar proxy is imperfect but already-built and non-crypto.

**Tier 3 — Raw-untrusted.** Unrecognized domains / no track record bind to nothing (`AnchorClass.BARE_KEY`, unchanged: weight 0.00, repCap 0.05). This is an existing bucket doing new work, not new code.

**Ingestion wiring (the one real behavior gap found in existing code):** `makeObservedStrand` (`api.ts` ~576–601) today mints `LIVE`/`WARM` unconditionally, regardless of source trust. A `writeFact` stamped `PUBLISHER_UNVERIFIED`/`PUBLISHER_TRACKED`/`BARE_KEY` must instead land as `fact_state: PROVISIONAL`, `tier: WARM` — reusing the **existing** `FactState.PROVISIONAL` semantics verbatim, no new enum value. Promotion to canonical status then goes through the **already-built** `ratify()` / `adjudicate(attr, {highImpact})` gates (decisive margin 0.30, min winner reputation 0.20, plus the ≥2-independent-root/recency/≥2-disjoint-class high-impact bar) — composing existing machinery, not building a parallel promotion path.

### 4.2 The multi-agent relay fix (the one genuinely new correctness requirement)

**The confirmed bug**, found independently by the DB expert and re-verified by the senior dev by reading `api.ts` directly: `provenanceRootFromStamp` (~line 543) mints `independenceClass = class:${stamp.source_id}` from the filing caller's identity alone. `#deriveAgreementSet` (~line 757) matches only on entity + `content_hash` + `LIVE`, with no derivation/relay check. `#R` (~line 780) unions provenance roots and hands them to `independentRootCount`. Result: Agent A researches a fact and writes it; A tells B in-context; B writes the same fact under its own stamp; because `stampA.source_id !== stampB.source_id`, the system mints two distinct independence classes and reports **2 independent roots from one relayed observation** — manufactured corroboration, with zero attacker involved. This is real, and it is not something the crypto machinery ever prevented (Ed25519 proves A and B are genuinely distinct *keys*; the bug is upstream of identity, in what counts as a "root" at all).

**The design adopted (senior dev's hardening of the DB expert's original proposal):**

1. `WriteFactInput` gains an **optional, additive** field:
   ```
   type CausalOrigin =
     | { kind: 'TOOL_CALL' | 'DOCUMENT'; resourceId: string }   // canonicalized resource identity
     | { kind: 'USER_STATEMENT' }
     | { kind: 'AGENT_RELAY'; consultedStrandIds: readonly StrandId[] };
   ```
   `resourceId` must be the canonicalized underlying resource (normalized URL / eTLD+1+path, or a content-hash of the fetched document) — **not** the ephemeral tool-call invocation id, correcting an under-collapse in the DB expert's original phrasing: two agents independently hitting the same search result should collapse to one real-world fact fetched twice, not two roots.

2. `independenceClass` computation changes: `AGENT_RELAY` **copies** the class(es) from the consulted strand(s)' existing provenance roots rather than minting a fresh one; `TOOL_CALL`/`DOCUMENT` hashes `(anchorClassId, resourceId)` so the same resource collapses regardless of which agent fetched it; `USER_STATEMENT` or omitted **falls back to today's behavior** (`class:${source_id}`) — the fail-closed direction, matching the codebase's established philosophy (the gap list already documents `independentSources` being corrected from fail-open to fail-closed for exactly this class of danger).

3. `writeFact` mints the `DERIVATION` edge **itself**, atomically, inside the same `withTxn` envelope it already uses, whenever `causalOrigin.kind === 'AGENT_RELAY'`. This is a deliberate correction of leaving edge-minting to "the orchestration/librarian layer as policy" (the DB expert's own objection: *"a librarian that forgets to mint the edge silently reopens the hole"*). Folding it into the engine verb is a **structural**, not policy, defense — matching CLAUDE.md's own stated design philosophy verbatim.

4. Because independence class is now collapsed **at write time**, `#deriveAgreementSet`/`#R` need **no new graph-reachability check** — the existing Bron–Kerbosch `independentRootCount` does the whole job unmodified. This is a genuine simplification over the DB expert's original proposal (which called for a runtime DERIVATION-reachability exclusion at agreement-check time): reuse of the exact-not-approximate MIS machinery instead of a new, depth-ambiguous graph walk.

5. **Named residual, not silently accepted:** both the relay-fix (causal-origin honesty) and the fleet-cap pricing (§4.1) bottom out at the **same** trust boundary — the orchestration layer around each agent telling the truth about what it consulted and which fleet/tenant it belongs to. A compromised or dishonest agent integration defeats both simultaneously; they are not two independent layers of defense, and the design doc should say so plainly rather than implying redundant safety nets.

### 4.3 Prompt injection — named, not solved here

Confirmed by direct design review (no code in this repo currently executes recalled content as instructions) and by the researcher's check of Zep/Graphiti's and mem0's own docs (silent on this): the reputation/independence machinery above governs how much **weight** a claim carries and whether it can overturn other claims. It says nothing about whether an agent reading a recalled strand's payload can be manipulated by adversarial text embedded in that payload (a scraped page containing "ignore prior instructions..."). This is a **control-flow / prompt-hygiene problem at the consumption boundary** — how the orchestration layer frames recalled payload to the LLM (always as clearly-delimited untrusted data, never as instruction text) — structurally outside what a provenance/independence layer can fix. It is flagged here explicitly, the same way CLAUDE.md already scopes out KYC/hardware binders as a named, deliberate non-goal, not silently assumed solved.

## 5. Operational Reality

**Org-boundary mapping.** Departments/business units inside one SSO tenant default to **one** `operatorClassId` (the tenant) — not adversarial to each other, splitting finer is a configurable knob defaulting off, matching CLAUDE.md's existing "intended knob" framing for the anchor table. Verified/flagged external partners (Entra ID B2B guest `userType`, Okta federation realms — noted by the operator as asserted from general platform knowledge and not re-verified this pass, so **spot-check before treating as load-bearing**) map to the cheaper `EMAIL_OAUTH`-grade row rather than full organization weight, since the IdP itself is signaling "real identity, not fully vouched-for." Dev/staging/prod is explicitly **not** an identity axis — it's a deployment-topology concern (separate store files per environment), and folding it into `operatorClassId` would be a category error. Post-acquisition business units on unmerged directories should default toward **more**, not fewer, independence classes when infrastructure is ambiguous — under-collapsing is the safe failure direction (over-splitting only slows reputation build-up; false-independence is the costly error).

**Reviewer queue, not a single operator.** The existing `pendingLedger.ts` `approve()` gate (provenance verifies, not a disputed-member author, anchor-independent via `independentSources`) generalizes into a role-based routing table keyed on the **same** SSO group claims: an `ApprovalContext.callerGroups` field, checked against a routing table ("pricing disputes → Finance-OU on-call") as an **additional precondition layered before** the existing self-approval/independence gates — never replacing them, since a routed reviewer can still be a disputed author or anchor-correlated. `db.adjudicate(attr, {highImpact})` disputes route to a smaller, higher-trust group via a stricter routing-table entry — zero new identity machinery. A real review backlog will accumulate for genuinely ambiguous multi-class disputes; staffing/SLA is named as a required, out-of-scope operational item, the same honest way CLAUDE.md already names its other operational gaps, rather than left implicit.

**Audit trail — two genuinely different claims, and only one survives intact.**
- *"Show me the reasoning trail"* is fully solved and is a **real, substantive differentiator**: `content_hash`, provenance roots, `fact_state`, `DERIVATION` edges, the causal-origin field from §4.2, and the hash-chained PENDING/APPROVAL ledger together reconstruct a genuine narrative ("filed by agent X from search result Y, corroborated by two independently-anchored sources, promoted by reviewer C under routing rule R"). Neither mem0 (no provenance concept at all) nor Zep/Graphiti (an episode *pointer*, no independence weighting, no reviewer trail — see §6) can produce this.
- *"Prove no privileged insider quietly edited history"* is **not** solved by an unsigned local hash-chain, and this must be disclosed as bluntly as the senior dev states it: a SHA-256 chain proves internal *consistency* (tamper anywhere and every downstream hash breaks, `verifyChain` names the first broken seq), but anyone with filesystem write access to the store can rewrite the chain from that point forward, recompute every subsequent hash, and it will verify perfectly. The removed Ed25519 signing specifically prevented exactly that (forging a valid signature requires the private key). This is a materially weaker guarantee than the original "the money artifact" framing in CLAUDE.md's pillar 4, and the design doc should state it in exactly that sentence, not a softer one. Mitigation (crypto-free): dual-write the hash-chain to a second, access-segregated, infrastructure-provided-immutable sink (e.g. WORM object storage under a different service account) — this narrows but does not close the gap, and has not been checked against any specific regulatory framework (SOX/HIPAA); that would need a compliance specialist's review, not an engineering judgment.

**Cost/complexity verdict.** Genuinely simpler to operate than before: no key custody/rotation/revocation (SCIM deprovisioning already solves lifecycle); no Merkle-witness infrastructure to stand up (already unbuilt per the gap list, and now not needed at all); onboarding becomes "already has a company login" instead of a keypair-mint-and-DNS-01-bind flow this project maintained. Genuinely still harder than mem0/Zep: none of this (SSO mapping, relay-fix, publisher ladder, reviewer routing) exists as code yet; the relay-fix pushes real, unenforceable-from-inside-this-codebase discipline onto the orchestration/librarian layer; the reviewer queue is a new operational role/SLA a company must staff, something mem0/Zep externalize onto nobody because their LLM is sole arbiter (the exact risk this design refuses to accept, at a real staffing cost). **Segmentation, stated explicitly rather than pitched as a universal replacement:** an SSO'd company running a multi-agent fleet over untrusted research content, with real poisoning exposure as a first-class requirement, should choose Intelligent DB. A solo developer or small team with no IdP, no reviewer bandwidth, and no active threat model is better served by mem0 or Zep — this design is not competitive on integration speed for that segment, and the portfolio framing should say so.

## 6. How This Beats mem0 and Zep — Specifically, Not Rhetorically

**mem0** (verified via its own docs and DeepWiki's indexed source description, `mem0/memory/main.py`): a two-stage LLM pipeline — one LLM call extracts candidate facts, a second LLM call is handed the candidate plus its top-k vector-similarity neighbors and unilaterally picks ADD/UPDATE/DELETE/NONE. No source-identity, anchor, or independence concept anywhere in that decision path — the LLM sees only "new fact" plus "nearest existing memories by embedding similarity," never who asserted either one or whether they share a causal root. This means **invariant 1 is violated by construction**, not in an edge case: the model literally is its own witness for every memory mutation. (Two third-party benchmark figures reporting mem0 scoring ~14.6%–20% on conflict-resolution/detection metrics surfaced in research but were not independently re-derived from a verified primary methodology — cited here as reported-by-outsiders, not load-bearing.) This project's own measured 79–98% attack-success-rate against a real mem0 sidecar under data-poisoning tests (`docs/marketing/COMPARISON.md`) is architecturally predicted by this finding, not a surprising result: no defense exists in mem0's design to measure against.

**Zep/Graphiti** (verified against `arxiv.org/html/2501.13956v1`, the Graphiti GitHub repo, and its DeepWiki-indexed documentation): a genuinely strong bi-temporal knowledge graph — four timestamps per edge (`t_valid`/`t_invalid` for world-truth, `t'_created`/`t'_expired` for system-knowledge), edges marked invalid rather than deleted (architecturally close to demote-never-delete), and every node/edge keeps a pointer back to the raw episode that produced it (a real, auditable lineage). But direct queries against its documentation confirm, in its own words: *"the documentation contains no information about source trust scoring, source independence tracking, or corroboration weighting mechanisms"* and *"the system appears to process all episodes uniformly without differential handling based on source reliability."* Contradictions are resolved by an LLM comparing the new edge to related existing edges, with the docs stating Graphiti "consistently prioritizes new information" — i.e., **its own extraction/invalidation LLM is the final arbiter**, with recency as the tiebreak, and no human-in-the-loop override is documented. Zep's hosted `group_id` mechanism is a genuine, useful multi-tenancy primitive but is **scoping** (who can see this graph), not **trust** (how much to believe a fact inside it) — it does not do what independence classes do. Honest framing for the portfolio: *Zep solved the temporal/deletion half of the problem well; it did not attempt the identity/independence half at all* — and this redesign's entire contribution is that half.

Concretely, this design beats both on: (a) independence-weighted corroboration that a same-root relay or a syndicated-publisher flood cannot manufacture (§4.1–4.2, structurally absent from both competitors); (b) a quarantine tier for untrusted-source content (`PROVISIONAL`/`WARM` ingestion gated by `PUBLISHER_*`/`BARE_KEY` classes) that neither competitor documents; (c) a human-reviewable, routed dispute queue rather than sole-LLM-arbiter resolution; (d) a genuine multi-hop reasoning trail (§5) neither competitor's architecture produces.

## 7. Migration Plan and Effort Shape

**Survives untouched** (confirmed by direct reading, not inference): `identity/anchors.ts`, `identity/reputation.ts`, `identity/index.ts`'s facade logic (ports stay interfaces; only concrete implementations behind them swap), `core/types.ts` (except the additive `WriteFactInput.causalOrigin` field and new `AnchorClass` rows), `forgetting/consolidation.ts`, `ratification/corroboration.ts`, and the `DERIVATION`-BFS core of `ratification/disown.ts` (now doing double duty: audit lineage *and*, indirectly via propagated `independenceClass`, the relay fix).

**Real rewrites:**
- `identity/keys.ts` — deleted, replaced by a session-identity registry satisfying the same `KeyRegistryPort` shape.
- `identity/binding.ts` + `identity/anchorRegistry.ts` — signed-attestation binders deleted; replaced by three plain claim producers (SSO-claim with the corrected `SSO_TENANT_MEMBER` weight, not a reuse of `ORGANIZATION`/`VERIFIED_HUMAN`; publisher-domain, reusing the existing registrar/ASN operator-cluster lookup; bare-key unchanged). `verifyAttestation`'s signature check is dropped; the existing `bind()` path is promoted from fallback to primary.
- `ratification/merkleLog.ts` — signing and inclusion-proof math dropped (low-regret: never had live witnesses per the gap list); the operator or built-in hash-chain kept, optionally dual-written to a second sink.
- `ratification/pendingLedger.ts` — Ed25519 signing dropped, hash-chain kept, the RC-5 distinct-approver check becomes a session-identity + the existing `independentSources` check, with the weakened attribution guarantee (§5) documented explicitly in code comments, not just this doc.
- `api.ts` — `WriteFactInput.causalOrigin` (additive, non-breaking — no existing test needs to change for the type to compile); `provenanceRootFromStamp`/`independenceClass` computation per §4.2; `writeFact` mints `DERIVATION` edges atomically for `AGENT_RELAY` origins inside the existing `withTxn`; `makeObservedStrand` gains trust-tier-aware `fact_state`/`tier` per §4.1.

**New, additive-only:** an SSO-claim → `AnchorBinding` producer; a publisher-domain → `AnchorBinding` producer (thin, reuses existing operator-cluster code); a resource-identity canonicalizer feeding `causalOrigin.resourceId`; three new `AnchorClass` rows (`SSO_TENANT_MEMBER`, `PUBLISHER_UNVERIFIED`, `PUBLISHER_TRACKED`) added to the existing enum and `ANCHOR_TABLE`, weighted deliberately below the adjacent rows they resemble (§4.1).

**Adversarial test story** (matching the ~370-test / red-team bar this project already holds itself to — not a happy-path afterthought):
1. **Relay-laundering regression** — A writes F, B relays via `AGENT_RELAY` naming A's strand; assert `independentRootCount` stays at 1 and a `DERIVATION` edge exists. Directly targets the confirmed bug.
2. **Cheap-tenant-flood test** — N freshly-created `SSO_TENANT_MEMBER` sources with no verified custom domain attempt to out-vote a tenured `DOMAIN`/`PUBLISHER_TRACKED` incumbent via `adjudicate(..., {highImpact:true})`; assert DEFERRED regardless of N. Directly targets the SSO-weight correction in §4.1.
3. **Genuine cross-tenant / cross-publisher positive test** (anti-over-strictness, the owner's explicit concern) — two differently-rooted, genuinely disjoint sources corroborate the same finding; assert `independentRootCount = 2` and promotion succeeds.
4. **Publisher ownership/operator-cluster collapse** — N syndicated URLs behind one registrar/ASN collapse to 1 independent root (mirrors the existing `DOMAIN` fleet-cap test).
5. **Ledger-attribution honesty test** — a tampered `signerSourceId` in the now-unsigned chain is caught by `verifyChain()` only when tampering happens *without* live DB write access; explicitly assert (and comment) that a forged attribution from an authenticated-but-malicious write path is **not** caught — turning §5's disclosure into an executable, documented residual rather than prose.
6. **Atomic-compound extension** — a forced mid-op throw while `writeFact` mints an `AGENT_RELAY` `DERIVATION` edge rolls back fully (rides the existing `withTxn` helper already exercised by `atomicCompound.test.ts`).
7. **PROVISIONAL-on-ingest test** — a `PUBLISHER_UNVERIFIED`-stamped `writeFact` lands `PROVISIONAL`/`WARM`, cannot win an `adjudicate` against a `LIVE` incumbent, and only reaches `LIVE` via explicit `ratify()` from a genuinely independent source.

Sizing note: the untouched set (the load-bearing math) is large, which is why this migration is tractable — but tests 1 and 2 are the two the senior dev flags as not safe to under-scope, since both go directly to whether the redesign still satisfies the hard theorem, which is the entire reason this architecture exists over mem0/Zep-shaped alternatives.

## 8. Open Risks and Trade-offs

**Resolved tensions between panelists (stated, not hidden):**
- *SSO tenant weight*: the DB expert and researcher's initial "map any SSO group claim onto `ORGANIZATION`/`VERIFIED_HUMAN`" was corrected by the senior dev's `SSO_TENANT_MEMBER` split (§4.1) — adopted here because the original would have silently reintroduced the contradiction-bomb under a friendlier name. This is the single most consequential correction in this synthesis.
- *Relay-fix mechanism*: the DB expert's original "exclude candidates reachable via a `DERIVATION` edge at agreement-check time" is superseded by the senior dev's "propagate causal origin into `independenceClass` at write time" (§4.2) — a strict simplification (reuses the existing exact MIS instead of adding a new, depth-ambiguous graph walk), adopted as the design of record. The DB expert's version is not wrong, just more expensive and less precise.
- *Audit non-repudiation*: the DB expert's and operator's framing ("the crypto-averse move," "mechanically the swap") understated this as a routine substitution. The senior dev's framing — a genuine security regression, stated in one blunt sentence — is adopted as the standard this doc holds itself to, matching the project's existing honesty bar.

**Unresolved, requiring the owner's decision:**
- Should the ledger-attribution regression (§5) be accepted as-is given the "no signing" constraint is explicit and absolute, or does it warrant a narrower carve-out (e.g., a deployment-keyed HMAC, arguably still short of "cryptography" in the forbidden sense) — a risk-tolerance call, not an engineering one.
- Should `SSO_TENANT_MEMBER` require a verified custom domain before exceeding `EMAIL_OAUTH`-grade weight (as proposed), or is there a cheaper-but-still-real-world-costly alternative signal (billing history length, verified business email domain match)?
- Is `causalOrigin`/`resourceId` capture realistically achievable across real agent frameworks, many of which interleave freeform tool calls without a structured citation list — and if an agent cannot cleanly report what it consulted, should the fallback default fail-closed (treat as a fresh independent root — conservative, safe against Sybil, under-counts genuine independent re-derivation) or something else? This is a real adoption-risk not fully closed by the design.
- Should the `PUBLISHER` anchor's operator-cluster axis ship at MVP using the reused registrar/ASN lookup alone (cheap, already-built, gameable by deliberately-obscured ownership per the researcher's own caveat), or is a curated media-ownership table a hard launch requirement for the research-content use case specifically?
- Should reviewer routing be static operator config (simpler to ship) or auto-derived from IdP group-naming conventions (scales better, assumes a convention this project can't guarantee)?
- Does the WORM dual-sink mitigation for the audit gap need to actually exist before the design can credibly be called compliance-ready in the portfolio framing, or is it acceptable as a named residual the way the rest of the gap list already operates?

**Standing, structural caveat carried forward from the project's own existing standard** (PRICED_ALTERNATIVES.md's "relocates, doesn't eliminate" framing, reapplied here): SSO delegation moves PKI to the IdP's OIDC/JWT stack; publisher/operator lookups sit on DNSSEC/TLS-secured infrastructure; TLS underlies every network call this design makes. "No cryptography anywhere in the system" is not a literally achievable end state — only "no cryptography this codebase builds, owns, or is the single point of failure for." That is the owner's actual stated bar, and this design meets it; it should never be described as more than that.
