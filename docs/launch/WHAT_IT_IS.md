# What Intelligent DB Is / What It Is Not

## The two failures it targets

AI agents fail in two distinct ways, and most memory tooling only addresses one. They **forget**: a stateless model loses everything outside its context window, so anything useful from five minutes or five sessions ago has to be re-derived or re-supplied. And they **hallucinate**: they invent facts, and — worse — they cannot tell the difference between something they actually recalled and something they just generated. A vector database with a bigger context window fixes the first failure mode somewhat and does nothing for the second: cosine similarity has no concept of "this fact came from a trustworthy, independent source" versus "this fact is a plausible-sounding fabrication." Intelligent DB is built to address both failure modes at once, by making provenance a first-class citizen of the storage model rather than an afterthought bolted onto retrieval.

## Memory palace, not nearest-neighbor

The central design choice is an inversion of how a vector DB works. A vector DB embeds everything into a shared space and, at query time, does fuzzy nearest-neighbor lookup across the whole corpus — every stored fact is always "in view," one similarity computation away. Intelligent DB does the opposite: facts (**strands**) sit latent by default, connected to each other only by **structural threads** — shared entity references and confirmed relationships, not embedding proximity. Nothing is a readable list. A cue energizes a seed strand, and **spreading activation** propagates outward across the strand graph, decaying with each hop (`child = parent * (w_edge / Σw_out) * γ`), until a bounded cluster of genuinely relevant strands "lights up." Only the lit cluster gets assembled into an answer.

This is the "memory palace / spider web" model: dense local webs (one conversation, one tight topic cluster) are sparsely bridged to other webs by rare **cross-web bridge** threads, and it's activation jumping a bridge that produces the "something from last week is suddenly relevant" effect a vector DB's flat similarity search can't reproduce on its own. The practical consequence is that retrieval quality is bounded by how well the graph was built (the "librarian" — the AI model deciding what connects to what) rather than by walk parameters; a good build makes recall structural, not statistical. Head-to-head numbers against tuned vector/hybrid retrieval, and the exact oracle-conditional caveats on those numbers, belong in the benchmark-grounded comparison doc this same pass produces (see "See also" below), not repeated here without their supporting methodology.

Facts also carry an explicit state machine — `LIVE`, `PROVISIONAL` (a visible superposition awaiting ratification), `DEMOTED` (was true, kept as history, never deleted) — and a hard distinction between **observed** facts (came from outside: a user, a document, a tool) and **derived** facts (the web computed them by walking itself). A derived fact can be believed and spoken, with its derivation shown, but it never counts as its own confirmation. This is the "wall with a window": a derived fact graduates to observed status only when an external source ratifies it.

## Two governing invariants

Everything else in the design falls out of two rules that are treated as non-negotiable:

1. **The model is never its own witness.** It files and speaks memories; it never confirms them. No provenance, no voice.
2. **The web is never its own witness about source identity.** Trust in *what* a fact is can be judged from inside the graph (contradiction, corroboration, recency). Trust in *who* is really behind a source cannot — a multi-agent adversarial review proved this formally: under "identity is priced, not prevented" (an attacker can pay a finite cost to mint independent-looking sources), no purely internal rule can simultaneously let one true witness overturn a planted false canonical fact *and* stop two fake sources from overturning a true incumbent. Those requirements are in direct tension, so identity has to be witnessed from outside the graph.

## The four pillars, at a glance

1. **Crypto-free trust registry** (`identity/trustRegistry.ts`, `identity/sources.ts`). Identity is *consumed* from infrastructure the deployment already trusts, never manufactured or cryptographically proven by this codebase: the owner's own word (the personal preset's ground truth), SSO tenant membership (deliberately priced email-grade unless the registry config verifies the tenant's custom domain), web publishers collapsed to their registered domain via a bundled Public Suffix List (fifty pages from one site count as one witness), and registry-configured systems of record. Independence between two sources is computed as an exact maximum-independent-set over anchor-disjointness (Bron–Kerbosch/Tomita), and an operator fleet axis collapses N domains behind one operator toward a single independence class instead of counting N times. (An earlier version of this layer built its own Ed25519 passports and DNS-01 binders; that machinery was deleted in the crypto-free rebuild — see `docs/launch/REBUILD_SUMMARY.md`.)
2. **Tamper-evident checksum-chain audit ledger** (`ratification/pendingLedger.ts`). Every audit record's SHA-256 checksum chains from the previous record; `verifyChain()` recomputes the whole chain and names the *first broken record* if any persisted byte was flipped, and `chainHead()` exports an O(1) checkpoint that a rewritten history cannot reproduce. For insider-tamper detection, an optional real-time `AppendSink` hook ships every record to external storage *before* the local write, so a rewrite diverges from the already-shipped copy at exactly the forged record (reference sink implementations in `src/examples/auditSinks.ts`). The deliberate, disclosed trade-off: attribution (*who* wrote a record) is asserted, not signed — detection against an insider rests on the shipped copies living somewhere the writer cannot touch. (The crypto-era Ed25519-signed Merkle layer this replaced is described, historically, in `docs/history/ARCHITECTURE.md`.)
3. **Beta(α,β) trust scoring** (`identity/reputation.ts`). Reputation is earned slowly (independence-weighted α) and lost fast (4× asymmetric β on contradiction), decays on a 90-day half-life applied at read time, is read out as a lower-confidence bound (a fresh source reads as exactly 0), and is capped by the anchor class it's rooted in. This is what defeats the "contradiction bomb" (500 fresh Sybil sources are individually plausible but collectively weightless) and the "first-arrival trap" (canonical status follows earned trust, not who spoke first).
4. **Undo-engine / disown sweep** (`ratification/disown.ts`). Disowning a source runs a BFS taint-closure over derivation edges that demotes (never deletes) everything downstream, reverses exactly the reputation credit that source's strands earned elsewhere, and can reopen an adjudicated dispute if removing the tainted evidence collapses its decisive margin. Demotion is archival, not destructive — the archive stub (content hash, provenance roots, timestamps) is immortal.

Beyond storage, the engine also exposes an MCP (Model Context Protocol) server (`src/mcp/server.ts`, with a `bin` entry in `package.json`) offering four tools — `remember`, `recall`, `list_pending_questions`, and `resolve_pending` (the personal-tier dispute horn) — so an agent host (e.g. Claude Code) can use Intelligent DB as a memory tool directly rather than only as an embedded library.

## What it is NOT

- **Not a vector database.** There is no embedding space and no nearest-neighbor search as the retrieval primitive. Similarity search can sit alongside it, but the substrate's actual memory model is a provenance graph traversed by spreading activation, not a similarity index.
- **Not a blockchain, and not cryptography.** There is no distributed consensus, no mining, no token, no peer-to-peer network — and, since the crypto-free rebuild, no keys, no signing, and no attestations anywhere in the codebase. The "chain" is a local, single-writer, SHA-256 checksum chain over an append-only SQLite ledger (plus an optional real-time shipping hook) — a tamper-*evidence* pattern, not a decentralized settlement layer or an identity scheme.
- **Not a consensus or majority-vote system.** Contradictions are never resolved by headcount. A same-class echo dispute resolves by reputation/anchor-cost/stake tiebreak; a genuine multi-independence-class dispute either auto-resolves on a decisive, earned reputation margin or defers to a human reviewer through a signed pending-ratification ledger. Flooding the graph with fresh, cheap, agreeing sources does not manufacture a winning vote — this is a deliberate structural defense against the contradiction-bomb attack, not an oversight.
- **Not a finished product.** It is a single-process prototype: TypeScript on Node (`>=20`, developed on Node 24), SQLite/WAL persistence, atomic compound writes, `npm test` currently green across the full Vitest suite (see CLAUDE.md for the current count — CLAUDE.md's Status line is the single canonical number), zero runtime dependencies (`package.json` has no `dependencies` key, only `devDependencies` powering the benchmark harness).

  Named, acknowledged gaps, straight from CLAUDE.md's own gap list: no cross-process/concurrent-writer support (SQLite WAL assumes one writer); audit attribution is asserted, not signed (the named crypto-free trade-off — insider-tamper detection needs the `AppendSink` / `chainHead()` copies stored where the writer can't reach, a deployment step); trust-registry claims are configuration, not proof (a misconfigured registry silently mis-weights independence); independence-class assignment is an offline human judgment call the code cannot verify; there is no encryption-at-rest, no backup/PITR, and no schema-migration ladder.

  Two findings an earlier draft of this document reported as open are now closed: the launch code review's `ratify()`-atomicity and `independentSources()`-fail-open findings were both fixed and regression-tested — `ratify()` now runs inside the same `withTxn` transaction helper as every other compound operation, and an unregistered source now fails *closed* in the approval gate's independence check. See [`docs/launch/BUGFIX_REPORT.md`](./BUGFIX_REPORT.md) for the per-fix diff summary, the discriminating regression tests, and the adversarial re-verification verdicts.

  None of this is hidden — it is the explicit, itemized residual of a design whose central claim is that Sybil resistance is *priced*, not *solved*: a patient, well-funded attacker who buys genuinely distinct, genuinely independent anchors still gets genuinely independent identities. The system's job is to make that the only path left, not to make it impossible.
